# Contract: Rust/Tauri Configuration Changes

**Issue:** THU-302
**Affects:** `src-tauri/Cargo.toml`, `src-tauri/src/lib.rs`, `src-tauri/capabilities/default.json`, `src-tauri/capabilities/auth.json`

---

## 1. `src-tauri/Cargo.toml`

### Add Dependency

Add to the `[dependencies]` section (alphabetically, after `tauri-plugin-opener`):

```diff
  tauri-plugin-opener = "2.5.2"
+ tauri-plugin-oauth = "2"
  tauri-plugin-shell = "2.3.3"
```

### No Feature Flag Needed

The OAuth plugin is always-on for desktop builds (no conditional compilation). It has no feature flags of its own. It does not need to be in the `[features]` section.

## 2. `src-tauri/src/lib.rs`

### Add Plugin to Builder Chain

Add `.plugin(tauri_plugin_oauth::init())` to the `create_app()` builder chain. Place it after `tauri_plugin_deep_link::init()` (since deep-link handles mobile OAuth, and this plugin handles desktop OAuth -- logical grouping):

```diff
      .plugin(tauri_plugin_deep_link::init())
+     .plugin(tauri_plugin_oauth::init())
      .plugin(tauri_plugin_updater::Builder::new().build())
```

### No Commands Needed

The plugin registers its own IPC commands (`plugin:oauth|start`, `plugin:oauth|cancel`) internally via `tauri::generate_handler!` inside the plugin's `init()`. No additions to the app's `invoke_handler` are needed.

### No State Management Needed

The plugin does not use Tauri managed state. Each `start()` call spawns an independent TCP listener thread. The port number serves as the handle for `cancel()`.

## 3. `src-tauri/capabilities/default.json`

### Add OAuth Permissions

Add to the `permissions` array. Place after `deep-link:default` for logical grouping:

```diff
      "deep-link:default",
+     "oauth:allow-start",
+     "oauth:allow-cancel",
      "updater:default",
```

Note: The plugin does NOT define an `oauth:default` permission set. The two commands must be listed individually. This was verified by inspecting the plugin's `permissions/autogenerated/` directory, which only contains `commands/start.toml` and `commands/cancel.toml`.

### WebviewWindow Permission Removal

Codebase analysis shows two consumers of webview permissions:

1. **OAuth (being removed):** Uses `new WebviewWindow(label, options)` -- requires `core:webview:allow-create-webview-window`
2. **Sidebar content preview (`src/content-view/use-sidebar-webview.ts`):** Uses `new Webview(window, label, options)` -- requires `core:webview:allow-create-webview`, `allow-set-webview-size`, `allow-set-webview-position`, `allow-webview-close`

**Only remove** `core:webview:allow-create-webview-window`. This permission creates a NEW window containing a webview and is only used by the OAuth flow.

**KEEP** all other webview permissions -- the sidebar content preview feature needs them:
```json
"core:webview:allow-create-webview",
"core:webview:allow-webview-size",
"core:webview:allow-webview-position",
"core:webview:allow-set-webview-size",
"core:webview:allow-set-webview-position",
"core:webview:allow-webview-close"
```

**KEEP** all window permissions (native OS windows, not webviews):
```json
"core:window:allow-create",
"core:window:allow-close",
"core:window:allow-hide",
"core:window:allow-show",
"core:window:allow-set-focus",
"core:window:allow-set-skip-taskbar",
"core:window:allow-destroy"
```

## 4. `src-tauri/capabilities/auth.json`

### DELETE This File

Current contents:
```json
{
  "$schema": "../gen/schemas/desktop-schema.json",
  "identifier": "auth",
  "description": "Strict, no-IPC capability profile for OAuth authentication windows",
  "windows": ["oauth-*"],
  "permissions": [
    "core:default",
    {
      "identifier": "http:default",
      "allow": [
        { "url": "https://accounts.google.com" },
        { "url": "https://www.googleapis.com" }
      ]
    },
    "deep-link:default"
  ]
}
```

This capability profile was created exclusively for `oauth-*` WebviewWindows. With the loopback approach:
- No WebviewWindows named `oauth-*` are created
- OAuth HTTP traffic (to `accounts.google.com`, `googleapis.com`) goes through the system browser, not through Tauri's HTTP plugin
- The `http:default` allowlist for Google URLs in `default.json` already covers any API calls the app makes directly (e.g., `getUserInfo`)

Delete the file entirely. No replacement is needed.

## 5. NPM Package

In addition to the Rust crate, the JavaScript guest bindings must be installed:

```bash
bun add @fabianlars/tauri-plugin-oauth@latest
```

This provides the `start`, `cancel`, `onUrl`, and `onInvalidUrl` functions imported in `oauth-loopback.ts`.

## 6. Google Cloud Console Configuration

The following redirect URIs must be registered in the Google Cloud Console OAuth 2.0 client configuration:

- `http://localhost:17421`
- `http://localhost:17422`
- `http://localhost:17423`

Google allows `http://localhost` redirect URIs for native applications (per RFC 8252, Section 7.3). No HTTPS is required for localhost.

## 7. Microsoft Azure AD Configuration

The following redirect URIs must be registered in the Azure AD app registration:

- `http://localhost:17421`
- `http://localhost:17422`
- `http://localhost:17423`

Microsoft's platform configuration for "Mobile and desktop applications" supports localhost redirect URIs.

## 8. Verification Steps

After making all Rust/Tauri changes:

1. Run `cargo check` in `src-tauri/` to verify the dependency resolves and `lib.rs` compiles
2. Run `cargo build` to verify full compilation
3. Run the app and verify no capability errors in the console
4. Verify `oauth:allow-start` and `oauth:allow-cancel` appear in the generated permissions (check `src-tauri/gen/` after build)
